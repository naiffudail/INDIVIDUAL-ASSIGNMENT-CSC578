Why Git

Git is a distributed version control system. Every clone of a repository is a full copy of the project's history, not a thin pointer back to a central server. That single design decision shapes almost everything else about how the tool behaves.

Because history lives locally, you can commit, branch, and inspect logs on a plane with no signal. Network access only matters the moment you want to share work with someone else, via push or pull.

Git tracks content, not files in the way you might expect. Internally, it stores snapshots of your entire project as a graph of objects, and it is extremely good at noticing when two snapshots share most of their data, even across renamed files.

  • Speed — almost every operation is local disk I/O.
  • Integrity — every object is addressed by a SHA-1 (or SHA-256) hash of its contents.
  • Non-linear workflows — branching and merging are cheap, so they're used constantly rather than avoided.
Getting Started

A repository begins with a single command run inside a project folder:

git init

This creates a hidden .git directory that holds the entire object database, the staging area, and configuration for that project. Nothing outside that folder is touched.

Before anything is committed, it helps to tell Git who you are, since every commit is permanently signed with this identity:

git config --global user.name "Ada Lovelace"

git config --global user.email "ada@example.com"

To pull down an existing project instead of starting fresh, use git clone followed by a URL or local path. Cloning copies the full history, not just the latest snapshot.

Snapshots and Commits

Git organizes work around three areas: the working directory where you edit files, the staging area (also called the index) where you assemble the next snapshot, and the repository where finished commits live permanently.

Moving a change from the working directory into the staging area is done with:

git add <file>

Once staged, a commit seals the snapshot with a message explaining the change:

git commit -m "Add login validation"

A good commit message describes intent, not mechanics. "Add login validation" tells a future reader why the change exists; "edited file.js" does not.

To see exactly what has changed before staging, git diff compares the working directory against the last commit, while git status gives a higher-level summary of what's tracked, modified, or untracked.

  • git log walks backward through commit history.
  • git show <commit> displays the full contents of one commit.
Branching

A branch in Git is nothing more than a movable pointer to a commit. There is no separate copy of the project sitting behind it, which is why creating one is nearly instantaneous regardless of project size.

Create and switch to a new branch in one step:

git switch -c feature/payments

The default branch, often named main, is just a convention, not a special object. Teams frequently keep it protected and require all work to land there through a reviewed merge.

Listing branches shows where each pointer currently sits:

git branch -v

Because branches are cheap, it's common to create one for every bug fix, experiment, or feature, then discard it the moment it's no longer needed.

Merging

Merging brings the history of two branches back together. Git looks for the most recent commit they share, then combines everything that happened on each side since that point.

When the changes don't overlap, Git resolves the merge automatically and creates a new commit with two parents:

git merge feature/payments

When two branches edit the same lines, Git pauses and marks the file with conflict markers, leaving the decision to a human. Resolving a conflict means editing the file to the correct final state, then staging and committing it like any other change.

An alternative to merging is rebasing, which rewrites one branch's commits on top of another to produce a straight line of history instead of a fork. It achieves a similar end result through a very different mechanism, and rewrites commit hashes in the process.

  • Fast-forward merge — no divergent history, the pointer simply moves forward.
  • Three-way merge — both branches moved, so a new merge commit ties them together.
Remotes

A remote is simply another copy of the repository, usually hosted on a server, that your local repository knows how to talk to by name. The default name for the remote you cloned from is origin.

Sending local commits to a remote branch:

git push origin main

Fetching commits from a remote without merging them into your current branch:

git fetch origin

git pull is the combination of a fetch followed immediately by a merge (or rebase, depending on configuration) into the branch you're currently on.

It's entirely possible, and common in larger projects, to track multiple remotes at once — for instance one for the official project and one for your own fork.

Undoing Things

Git keeps almost everything recoverable, but the commands for undoing work are sharp tools and worth understanding before reaching for them under pressure.

To unstage a file without losing the edits in your working directory:

git restore --staged <file>

To discard local changes entirely and return a file to its last committed state:

git restore <file>

Rewriting the most recent commit, often to fix a typo in the message or add a forgotten file, is done with:

git commit --amend

Moving the branch pointer backward with git reset can rewrite history, so it's best reserved for commits that haven't been pushed anywhere yet. Once shared, history is best appended to rather than rewritten.
